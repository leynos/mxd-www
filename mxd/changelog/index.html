Changelog

Release notes follow the compatibility matrix governance process: every release that touches client-facing behaviour includes a QA sign-off statement linking to the internal compatibility matrix. If the release introduces a new compatibility quirk, the deviation is recorded in both the release notes and the matrix.

v0.4

Version 0.4.0

2026-03-10

Stable Breaking

Added

  • Transaction routing layer for Login (0x006B) with full XOR detection and decoding
  • Hexagonal architecture implementation with ports and adapters for protocol boundary isolation
  • Property-based testing suite using proptest for frame validation
  • TLA+ specification for handshake state machine with liveness proofs

Changed

  • Breaking: Codec API now returns Result instead of Option
  • Handshake validation now rejects protocol ID mismatches before allocating session state
  • Multi-fragment reassembly buffer now uses bounded capacity with explicit overflow handling

Fixed

  • Byte order handling for transaction ID field on big-endian systems
  • XOR decoding now correctly handles zero-length text fields
  • Frame header validation no longer panics on malformed total-size field

QA Sign-off

Compatibility matrix updated with XOR detection quirk for Hotline Client 1.2.3. All regression tests pass against reference client implementations. See ADR-004 for codec error handling rationale.

Release Governance

Every release that modifies client-facing behaviour requires a QA sign-off statement. The statement must:

  • Link to the updated compatibility matrix
  • Identify all reference clients tested against the release
  • Document any new compatibility quirks or deviations
  • Reference relevant ADRs for architectural decisions

Version Numbering

mxd follows semantic versioning with the following interpretation:

  • Major (0.x.0): Breaking changes to codec API or protocol implementation
  • Minor (0.x.0): New transaction types or feature additions
  • Patch (0.0.x): Bug fixes and internal refactoring

Changelog Sources

Release notes are generated from Git tags and maintained in CHANGELOG.md. The compatibility matrix lives in docs/compatibility/ and is versioned alongside releases.
